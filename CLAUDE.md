# VenueDash ‚Äî Project Context

VenueDash is a SaaS platform for Atlanta studio owners who rent their spaces for private events. It handles the paperwork layer of event rentals: signed contracts, timestamped condition-photo walkthroughs, and damage-deposit status tracking.

**What exists and works today:** an owner signs up (Clerk) ‚Üí 5-step wizard at `/settings` (doubles as Settings forever) ‚Üí live `/book/[slug]` link on the dashboard. A renter (no account) opens `/book/[slug]`, picks an available date/time, submits intake ‚Üí a `pending` booking is created (terms snapshotted), owner + renter receive emails, and the renter lands on a durable `/status/[token]` page. Landing at `/` with waitlist (Resend contacts). **The owner now acts in-app (Phase 5):** a dark two-pane owner shell (sidebar over the `(owner)` group) with a grouped dashboard (`/dashboard` ‚Äî Needs action / In progress / Past + a money-free metric strip) and a booking detail (`/dashboard/bookings/[id]`: lifecycle rail + state-derived cards) that drives approve‚Üí`awaiting_contract` / decline‚Üí`declined` / cancel / mark-signed‚Üí`confirmed` via `transitionBooking`, plus a manual `deposit_status` control. Seeded dev DB: `npm run db:seed` (Westview Studio, slug **`westview`** + 10 bookings across all 9 states) ‚Äî **gotcha:** the seed studio is owned by a fixed `clerkUserId "seed-owner"` (`scripts/seed.ts`), so a real signed-in owner won't see the roster; repoint the `westview` studio's `clerk_user_id` to your Clerk id to walk it in the owner UI.

**v0.5 scope guardrails (supersede older sections wherever they conflict):**
- **No held deposits.** VenueDash never touches deposit money; the owner collects/refunds off-platform and VenueDash only records `deposit_status` (`uncollected|collected|returned`). Do not reintroduce VenueDash-held deposits without revisiting the spec.
- **COI collection, damage-claim flow, automated e-signature, and Stripe are OUT of v0.5** (v1.0). Contract signing is manual.
- The booking state machine is a trimmed subset of the v1.0 enum, same state names: `pending ‚Üí declined|awaiting_contract ‚Üí awaiting_signature ‚Üí confirmed ‚Üí event_day ‚Üí post_event ‚Üí closed`, `canceled` from pre-event states; `declined/closed/canceled` terminal.

## Architecture (as built)

**Stack:** Next.js 16 (App Router) ¬∑ React 19 ¬∑ Tailwind v4 (`@theme` in `app/globals.css`) ¬∑ Clerk 7 ¬∑ Drizzle + Neon Postgres (websocket Pool driver) ¬∑ Cloudflare R2 ¬∑ Resend. Route groups `(marketing)` / `(public)` / `(owner)`.

**Domain layer (the spine ‚Äî read before touching bookings):**
- `lib/domain/states.ts` ‚Äî `BOOKING_STATES`, `LEGAL_TRANSITIONS` (single source of truth).
- `lib/domain/transitions.ts` ‚Äî `transitionBooking(db, id, to, actor, opts?)`: transactional CAS + append-only `booking_events` audit + post-commit hooks. **No other code path ever writes `bookings.state`.** `booking_events` has no update/delete path, ever.
- `lib/domain/effective-state.ts` ‚Äî read-time clock-state derivation (`confirmed‚Üíevent_day‚Üípost_event`).
- `lib/domain/booking-view.ts` ‚Äî **pure view-model for the owner UI:** `toBookingView(booking, now)` ‚Üí effective state, dashboard group (needs_action/in_progress/past), legal owner actions (`cancel` suppressed once effective `event_day`/`post_event`), deposit-control flag, status chip. The single brain; owner RSCs stay thin. Unit-tested across all 9 states.
- `lib/tokens.ts` ‚Äî renter tokens: 32-byte base64url raw, SHA-256 hash at rest, one active per `(booking, purpose)`, re-mint rotates. `createBooking` mints `purpose="status"` at creation; `/status/[token]` renders read-only booking state from it (durable, not single-use).
- `lib/booking.ts` ‚Äî `createBooking(db, input)`: the **sanctioned creation path**. `pending` is the schema-default *genesis* state, so this is a plain INSERT ‚Äî **not** a `transitionBooking` call ‚Äî and writes **no** `booking_events` row (`createdAt` is the creation record). Snapshots terms onto `rateSnapshot` (rate/minHours/ladder/policies/maxOccupancy) + `depositCents`. Also `getBusyIntervals` (non-terminal bookings + manual blocks) for conflict-checking. **Phase 5 owner reads/writes:** `listBookingsForStudio`, `getBookingForOwner` (**studio-scoped ownership boundary** ‚Üí `notFound()` on a foreign id), `getBookingEvents` (lifecycle rail), `setDepositStatus` / `setContractSignedAt` (plain column updates ‚Äî **not** transitions, write no `booking_events`).
- `lib/tz.ts` ‚Äî Atlanta (`America/New_York`) wall-clock ‚Üî UTC, DST-aware two-pass offset. `lib/availability.ts` ‚Äî pure interval math (`overlaps`/`hasConflict`/`availableStartHours`). `lib/rate-limit.ts` ‚Äî DB-backed fixed-window limiter (`checkRateLimit`, atomic upsert; `rate_limits` table).
- `lib/studio.ts` ‚Äî studio persistence (slug minted once, immutable; `onboarding_completed_at` set once); `getStudioBySlug`, `maxOccupancyOf`. `lib/money.ts` ‚Äî `parseDollarsToCents`, `formatCents`.
- All DB-touching functions take the Drizzle handle (`Db` from `lib/domain/transitions`) as their first parameter; PGlite tests inject `createTestDb()` from `lib/domain/test-db.ts`, which applies the **real** generated migrations from `drizzle/`.
- Cross-cutting: snapshot terms (rate/deposit/policies) onto the booking at request time; never re-join studio settings for legal fields.

**DB access:** `lib/db.ts` exposes lazy `getDb()`/`getPool()` (Neon **websocket** Pool + `ws`; imports must stay on the Node runtime ‚Äî never Edge). Nothing connects at import time (CI/builds have no DATABASE_URL). Auth middleware lives in **`proxy.ts`** (Next 16's renamed middleware, Node runtime ‚Äî do not create `middleware.ts`); its matcher gates `/dashboard(.*)` and `/settings(.*)` ‚Äî extend it when adding owner routes.

**Server-action pattern (established in Phase 3, follow it):** thin `"use server"` actions = Clerk `auth()` (owner) ‚Üí pure FormData parser (colocated `forms.ts`, unit-tested) ‚Üí `lib/*` persistence (PGlite-tested) ‚Üí `redirect()` on success / return form-state on error. **A `"use server"` file may only export async functions** ‚Äî a `const` exported from one reaches client components as a broken reference and crashes at render (this shipped as a bug once; constants live in plain modules like `forms.ts`). The public `/book/[slug]` submit follows the same shape with two front gates ‚Äî **honeypot** (silent bounce) then **IP rate-limit** (`checkRateLimit`, before any DB/email work) ‚Äî and **best-effort emails**: a `sendEmail` failure is caught and logged but never fails the booking, and `redirect()` stays outside every try/catch (it throws control-flow). Owner email address is fetched from **Clerk** (`clerkClient().users.getUser`), not stored in our DB. **Owner mutation actions (Phase 5)** follow the same shape but `revalidatePath` and **stay on the page** instead of `redirect()`; each re-resolves the studio from the Clerk `userId` and re-fetches the booking studio-scoped (`getBookingForOwner`) ‚Äî the client-supplied `bookingId` is never trusted for authz. New owner routes nest under `/dashboard/*` (already covered by the `proxy.ts` matcher ‚Äî no matcher change needed).

## Dev setup & workflow

- **Use Node 20** (`nvm use 20`; default shell Node is 24 and engine-strict rejects it). Prefix npm commands accordingly.
- Copy `.env.example` ‚Üí `.env.local` (Clerk, Neon `DATABASE_URL`, R2, Resend). Resend API key must be **Full access** (sending-only keys 401 on the Contacts API).
- Scripts: `npm run dev | build | lint | typecheck | test | db:generate | db:migrate | db:seed | db:healthcheck`. Tests run on PGlite ‚Äî no secrets needed; CI runs lint/typecheck/test/build.
- **Vercel:** framework pinned in `vercel.json`. The four `NEXT_PUBLIC_CLERK_*` redirect vars are baked at build time ‚Äî set on **both Production and Preview**; a cached "Redeploy" won't pick up changes, push a commit. **`APP_URL`** (server-only *runtime* var ‚Äî origin for transactional email links) is set on **Production only** (currently `https://venue-gold.vercel.app`); leave it **unset on Preview/dev** so the submit action falls back to the request host (the correct per-deploy URL). Because it's not `NEXT_PUBLIC_`, it's changeable without a rebuild (swap it when a custom domain lands).
- **Email deliverability (working via Resend's default sender):** booking/status emails are **best-effort** ‚Äî a send failure never blocks the booking, so it's silent; diagnose via the **Resend dashboard logs**. `EMAIL_FROM` is intentionally **UNSET in Vercel** (both scopes), so `lib/email.ts` falls back to `VenueDash <onboarding@resend.dev>` (Resend's shared verified sender) and sends **do deliver** ‚Äî verified 2026-07-06 to a public Gmail. **Footgun when re-adding `EMAIL_FROM`:** paste it UNQUOTED in the Vercel UI ‚Äî a value copied from `.env.example` keeps its surrounding double-quotes, reaches Resend as `"‚Ä¶"`, and 422s (`Invalid from`); this silently killed every send once. A branded `venuedash.com` sender additionally needs the domain verified in Resend (pre-launch checklist).
- **DB migrations:** `npm run db:migrate` (drizzle-kit, idempotent) applies `drizzle/*.sql` to the `DATABASE_URL` in `.env.local` ‚Äî the same Neon DB the deployment reads. Run it after any schema change and before deploying code that needs the new table (Phase 4 added `0002` = `rate_limits`).
- **Process per phase:** brainstorm (superpowers) ‚Üí spec in `docs/specs/` ‚Üí plan in `docs/plans/` ‚Üí subagent-driven development on a `feat/phase-N-*` branch ‚Üí whole-branch review ‚Üí PR with preview-deploy checks ‚Üí merge. Ledger in `.superpowers/sdd/progress.md`.
- **Verification lesson (hard-won):** signed-out curl checks are not enough ‚Äî every new owner/renter page must be **rendered** in verification (authenticated walk on the preview, or an unauthenticated debug-route render of the client components locally).

## Phase 6 as built (contract generation) ‚Äî read before Phase 7

- **Orchestration (`lib/contract.ts`):** `contractKey` (`contracts/{id}/agreement.pdf`, deterministic), `getContractForBooking`, `upsertContract` (one-per-booking via `onConflictDoUpdate` on the unique `booking_id`), `markContractSigned`, and `generateAndAdvance(db, booking, identity, deps, actor)` ‚Äî build‚Üírender‚Üíput‚Üíupsert‚Üí`transitionBooking(awaiting_signature)`; **`transitionBooking`'s CAS is the sole idempotency/race guard** (double-click ‚áí IllegalTransition, caught). Render/put are **injected deps** so the DB path is PGlite-testable without R2/react-pdf.
- **Wiring:** owner action `generateContract` (guards `awaiting_contract`, best-effort **link-less** renter email via `ContractReadyRenter`, `markSigned` now also flips the contract row to `signed`); `booking-view.ts` gained the `generate_contract` OwnerAction; presigned download routes at `GET /dashboard/bookings/[id]/contract` (studio-scoped) and `GET /status/[token]/contract` (status-token). Both routes + the action stay on the **Node runtime** (react-pdf/aws-sdk). New: `lib/storage.ts` `putObject`; `lib/tz.ts` `formatAtlantaRangeLong` (year-inclusive, for the contract; UI keeps `formatAtlantaRange`).
- **‚öÝÔ∏è Migration `0003` (pdf_r2_key + unique booking_id) is NOT yet applied to Neon** ‚Äî run `npm run db:migrate` before the preview walk/merge (auto-mode blocked the automated run; it's a human step).
- **Phase 6 carry-forwards / deferred:** signed-PDF upload (`contracts.signed_pdf_r2_key` stays null in v0.5 ‚Äî v1.0 when auto-e-sign lands); contract regeneration/versioning (terms change ‚áí cancel+rebook today); a `mapTransitionError` shared helper (the 3-error catch is duplicated in `generateContract`); the contract-ready email could deep-link if the action threaded the request origin; **attorney review of the template is still the open launch gate (spec ¬ß7).**

- **(historic, now done) Phase 6 filled the one open state edge: `awaiting_contract ‚Üí awaiting_signature`** ‚Äî mark-signed still drives `awaiting_signature ‚Üí confirmed` and stamps `contract_signed_at`.
- **The contract reads the snapshot, not live settings.** `bookings.rateSnapshot` is the full 8-field `TermsSnapshot` (rate/minHours/ladder/policies/maxOccupancy) + `depositCents`, snapshotted at request time; `scripts/seed.ts` was **aligned to that shape in Phase 5** (seeded rows now carry policies). Interpolate mechanically (names/dates/amounts); never re-join studio settings for legal fields; no clause selection by legal reasoning (spec ¬ß7).
- **Owner surface conventions (Phase 5 ‚Äî follow them):** state-derived via `lib/domain/booking-view.ts` (`toBookingView`); effective state via `effective-state.ts`, never raw `booking.state`; ownership enforced server-side via `getBookingForOwner`; owner actions `revalidatePath` + stay on page. Copy discipline: **"timestamped documentation," never "immutable evidence"**; no held-deposit/COI/claim language.
- **Deferred to v1.0 (spec ¬ß8):** the **"Waiting on renter" dashboard group** (`booking-view.ts` reserves the concept) returns when renter-side signing lands ‚Äî the expected **first post-v0.5 step**; **clock-state persistence + close-out** (`post_event ‚Üí closed` is only reachable for stored-`post_event` today, so "close out" was intentionally left out of Phase 5); deposit audit history.
- **Availability is TOCTOU-tolerant by design:** a `pending` request is not a reservation ‚Äî overlaps are allowed and the owner arbitrates at approval; hard exclusion (buffer blocks on confirm) is deferred. `availability_blocks` currently only holds `manual` blocks.
- **Future UI pass (human-requested):** the **deposit-status control** deserves a cleaner treatment (its own full-width row out of the cramped 3-col status grid ‚Äî the current vertical list fixes a card-overflow bug, not the final design); **"Agreed terms" shows raw policy enums** (add a `POLICY_LABELS` map); lifecycle-rail genesis-`pending` cosmetic; sidebar `aria-disabled` + `pathname.startsWith` prefix footgun; graceful owner-action **session-expiry** handling (a lapsed Clerk session on an open page surfaces a generic "unexpected response" on action click).
- **Older backlog (non-blocking):** `rate_limits` has no TTL cleanup (v1.0); `availableStartHours` lacks half-open-edge tests; IP rate limit trusts the first `x-forwarded-for` hop (fine for single Vercel proxy); gate the share-link card on `onboarding_completed_at`; `createStudio` unique-violation retry; "save and stay" settings UX.
- **Palettes:** renter/public surface = **warm-light** (`renter-*` tokens, Instrument Serif display, mobile-first); owner/marketing = **dark**.

- Renter/public surface (warm light): bg `#f7f5f0`, ink `#211f1a`, border `#ddd7c6`, ok `#4d7c4a`.

Don't put a width utility in a shared input class that flex-row items reuse (`w-full` overrides `flex-1`/`w-[90px]` ‚Äî this also shipped as a bug once).
