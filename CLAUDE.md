## Architecture (as built)

**Stack:** Next.js 16 (App Router) Â· React 19 Â· Tailwind v4 (`@theme` in `app/globals.css`) Â· Clerk 7 Â· Drizzle + Neon Postgres (websocket Pool driver) Â· Cloudflare R2 Â· Resend. Route groups `(marketing)` / `(public)` / `(owner)`.

**Domain layer (the spine â€” read before touching bookings):**
- `lib/domain/states.ts` â€” `BOOKING_STATES`, `LEGAL_TRANSITIONS` (single source of truth).
- `lib/domain/transitions.ts` â€” `transitionBooking(db, id, to, actor, opts?)`: transactional CAS + append-only `booking_events` audit + post-commit hooks. **No other code path ever writes `bookings.state`.** `booking_events` has no update/delete path, ever.
- `lib/domain/effective-state.ts` â€” read-time clock-state derivation (`confirmedâ†’event_dayâ†’post_event`).
- `lib/domain/booking-view.ts` â€” **pure view-model for the owner UI:** `toBookingView(booking, now)` â†’ effective state, dashboard group (needs_action/in_progress/past), legal owner actions (`cancel` suppressed once effective `event_day`/`post_event`), deposit-control flag, status chip. The single brain; owner RSCs stay thin. Unit-tested across all 9 states.
- `lib/tokens.ts` â€” renter tokens: 32-byte base64url raw, SHA-256 hash at rest, one active per `(booking, purpose)`, re-mint rotates. `createBooking` mints `purpose="status"` at creation; `/status/[token]` renders read-only booking state from it (durable, not single-use).
- `lib/booking.ts` â€” `createBooking(db, input)`: the **sanctioned creation path**. `pending` is the schema-default *genesis* state, so this is a plain INSERT â€” **not** a `transitionBooking` call â€” and writes **no** `booking_events` row (`createdAt` is the creation record). Snapshots terms onto `rateSnapshot` (rate/minHours/ladder/policies/maxOccupancy) + `depositCents`. Also `getBusyIntervals` (non-terminal bookings + manual blocks) for conflict-checking. **Phase 5 owner reads/writes:** `listBookingsForStudio`, `getBookingForOwner` (**studio-scoped ownership boundary** â†’ `notFound()` on a foreign id), `getBookingEvents` (lifecycle rail), `setDepositStatus` / `setContractSignedAt` (plain column updates â€” **not** transitions, write no `booking_events`).
- `lib/tz.ts` â€” Atlanta (`America/New_York`) wall-clock â†” UTC, DST-aware two-pass offset. `lib/availability.ts` â€” pure interval math (`overlaps`/`hasConflict`/`availableStartHours`). `lib/rate-limit.ts` â€” DB-backed fixed-window limiter (`checkRateLimit`, atomic upsert; `rate_limits` table).
- `lib/studio.ts` â€” studio persistence (slug minted once, immutable; `onboarding_completed_at` set once); `getStudioBySlug`, `maxOccupancyOf`. `lib/money.ts` â€” `parseDollarsToCents`, `formatCents`.
- All DB-touching functions take the Drizzle handle (`Db` from `lib/domain/transitions`) as their first parameter; PGlite tests inject `createTestDb()` from `lib/domain/test-db.ts`, which applies the **real** generated migrations from `drizzle/`.
- Cross-cutting: snapshot terms (rate/deposit/policies) onto the booking at request time; never re-join studio settings for legal fields.

**DB access:** `lib/db.ts` exposes lazy `getDb()`/`getPool()` (Neon **websocket** Pool + `ws`; imports must stay on the Node runtime â€” never Edge). Nothing connects at import time (CI/builds have no DATABASE_URL). Auth middleware lives in **`proxy.ts`** (Next 16's renamed middleware, Node runtime â€” do not create `middleware.ts`); its matcher gates `/dashboard(.*)` and `/settings(.*)` â€” extend it when adding owner routes.

**Server-action pattern (established in Phase 3, follow it):** thin `"use server"` actions = Clerk `auth()` (owner) â†’ pure FormData parser (colocated `forms.ts`, unit-tested) â†’ `lib/*` persistence (PGlite-tested) â†’ `redirect()` on success / return form-state on error. **A `"use server"` file may only export async functions** â€” a `const` exported from one reaches client components as a broken reference and crashes at render (this shipped as a bug once; constants live in plain modules like `forms.ts`). The public `/book/[slug]` submit follows the same shape with two front gates â€” **honeypot** (silent bounce) then **IP rate-limit** (`checkRateLimit`, before any DB/email work) â€” and **best-effort emails**: a `sendEmail` failure is caught and logged but never fails the booking, and `redirect()` stays outside every try/catch (it throws control-flow). Owner email address is fetched from **Clerk** (`clerkClient().users.getUser`), not stored in our DB. **Owner mutation actions (Phase 5)** follow the same shape but `revalidatePath` and **stay on the page** instead of `redirect()`; each re-resolves the studio from the Clerk `userId` and re-fetches the booking studio-scoped (`getBookingForOwner`) â€” the client-supplied `bookingId` is never trusted for authz. New owner routes nest under `/dashboard/*` (already covered by the `proxy.ts` matcher â€” no matcher change needed).
